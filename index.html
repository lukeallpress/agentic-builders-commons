Why we exist

Most AI groups are still defining agentic AI. This is a practitioners' group: people moving past awareness into the practical work of building with agents — against real data, real privacy constraints, and real users — without creating avoidable risk.

Solve infrastructure together

Hosting, local/open-weight models, MCP servers, authentication, delegated accounts, role-based data access, and cost.

Make building safe

Training, guardrails, starter packs, and "builder's permit" pathways so staff can innovate without risking student data.

Share what works

Concrete recipes, code patterns, and use cases — the implementation-level ideas too technical for broad policy groups.

Ideas worth stealing

Reusable patterns and guardrails the group has surfaced. Take any of them.

Data Layer Pilot

MCP server with role-based data access

A Python MCP server sits between LLMs/agents and the district data warehouse (Redshift/PostgreSQL, PowerSchool) and enforces role-based access control, so a principal, teacher, cabinet member, or analyst only retrieves the rows and fields appropriate to their role. Agents ask approved tools rather than querying production databases directly.

MCPPythonRedshiftPostgreSQLPowerSchoolAWS
Governance Prototype

Builder's permit / responsible-builder pathway

A training-and-certification pathway staff must complete before publishing user-facing or data-facing tools. Builders demonstrate competence in FERPA/data privacy, authentication, secure deployment, validation, documentation, and maintenance. Permission to publish is tied to training and review; starter packs/boilerplates are provided for common build paths. Motivated by an incident where a well-intentioned champion built a dashboard that raised FERPA concerns.

Operations Pilot

Self-healing software pipeline

Claude Code monitors bug reports in the IT ticketing system, triages them, creates GitHub issues, and on a recurring schedule generates pull requests. Multiple models (Gemini, Copilot, Claude) review the PRs; an automation reviews the reviews; the system iterates until the issue is resolved.

Infrastructure Pilot

Delegated agent accounts (assistant model)

Each agent gets its own Google Workspace account with delegated access to a user's email and calendar — scoped like an executive assistant rather than an unrestricted clone of the user. Agents can draft emails but cannot send them.

Google WorkspaceGoogle ChatOpenClaw
Infrastructure Production

Segmented per-user micro-VM agent runtime

Each user's agent runs in its own AWS micro-VM (Agent Core), with per-user S3 storage, that terminates after ~30 minutes of inactivity and cold-starts (1-3 min) on the next interaction. AWS EventBridge and Lambda handle scheduling and prompt-triggered events ('remind me at X to Y'). Segmentation reduces cross-user contamination and controls cost.

AWSmicro-VMsS3EventBridgeLambdaOpenClaw
Data Layer Concept

Semantic / intelligence layer over the data warehouse

A semantic layer so agents correctly interpret business terms — e.g. what 'average daily attendance' or 'present' means — consistently across districts, without hardcoding values. Current approach uses field comments and a 'learnings' tool in the MCP server to capture context. Open challenge: how to maintain this at scale.

Workflow Concept

The four phases of an agentic workflow

An agentic workflow is a designed sequence, not just a better prompt: (1) Context Engineering — decide what the agent can see; (2) Prompt Engineering — define the job, boundaries, and stop conditions; (3) Tool Use — give only the tools needed; (4) Use/Evaluate — humans verify, decide, and own the action. Human design lives at the beginning; human judgment returns at the end.

Governance Concept

Narrow access + human review (the chef model)

The core design lesson across the group: choose the pantry, limit the appliances, and require review before serving. Give the agent the smallest set of evidence and tools needed, and keep a human accountable for the final action. 'Do not hand the agent a blowtorch when the task needed a flashlight.'

Operations Production

Automated multi-model code review

When a pull request is opened, three major models (Gemini, Codex, Copilot) review it as automated GitHub Actions, including FERPA/security/compliance audits from different angles. Claude Code then refines the code based on the feedback, reducing human review even for unfamiliar languages.

Data Layer Prototype

Learnings vs. skills for agent knowledge

Agents expose get-learnings / write-learnings tools so that when an LLM discovers something useful it records a 'learning' others can reuse — akin to a skill exposed by the MCP server. Open debate: is this the right schema, and how do you store knowledge scalably without overflowing the context window, versus building formal 'skills'?

Governance Concept

Read-only-first deployment ladder

Tools turn an answer engine into a workflow partner, so add capability in stages: Inspect (read files, search approved docs, query scoped data), then Draft (prepare emails, reports, tickets, code), then Act (send, update, route, publish, change records). Reserve real-world action for approved workflows.

Governance Concept

Vetted GitHub repository for shared agent code

Share the group's agentic code through a central, peer-audited GitHub repository so others can safely adopt and replicate it. Code is vetted via peer audits/reviews (security flaws, implementability) before it's recommended — modeled on practices like reviews.md files used at larger orgs.

Workflow Prototype

Agent skill colony (self-writing feedback loop)

A pattern adapted from software practice for curriculum design: multiple 'roles' (e.g. curriculum auditor, lesson renderer) review and modify their own skills based on the output, iterating until the system produces a successful result. Generated curriculum well, but hit limits on quality-control tasks like visual consistency.

Leadership Concept

Building hyper-local tools in-house vs buying

School systems can increasingly build their own tools in-house to save money and bake in local context that commercial vendors miss — enabling hyper-local solutions that weren't feasible a year ago. The group wants to push this conversation beyond product-specific ideas to its systemic implications.

Open questions we're chewing on

  • Investigating How do we store agent knowledge ('learnings') scalably without overflowing the context window?

  • Investigating How do we maintain a semantic/intelligence layer so agents interpret terms (e.g. 'average daily attendance') consistently across districts?
  • Open Is there a way to formalize this practitioners group within CoSN, or should it be a separate group?
